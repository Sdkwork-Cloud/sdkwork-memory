# MEMORY Database Module

Canonical lifecycle assets for `sdkwork-memory` under `DATABASE_FRAMEWORK_SPEC.md`.

- moduleId: `memory`
- serviceCode: `MEMORY`
- tablePrefix: `ai_`
- engines: PostgreSQL for production; SQLite for local development and tests

## Migration Authority

Schema changes are authored only in the application-root migration directories:

1. `database/migrations/{postgres,sqlite}/*.up.sql` with paired down migrations is the only DDL authority.
2. `database/ddl/baseline/{engine}/0001_memory_baseline.sql` is generated by folding the authoritative up migrations.
3. `plugins/sdkwork-memory-plugin-native-sql/migrations/` contains no SQL and is not an authority.

Run `pnpm db:materialize:baseline` after migration changes. `pnpm verify` checks that the generated baseline is current.

## Physical Profile

This pre-release module uses a migrations-only lifecycle for greenfield deployments.

- Every business-table `id` is an application-generated SDKWork Snowflake `BIGINT`; neither engine allocates row IDs.
- JSON and UTC instant logical values use validated `TEXT` in the native SQL profile. JSON is validated at application boundaries and is not queried with database JSON operators.
- Continuous confidence, ranking, and weight values use PostgreSQL `DOUBLE PRECISION` and SQLite `REAL`. They are not monetary or exact-decimal values.
- Outbox deliveries, learning jobs, and eval runs persist owner/token/expiry leases. Completion and acknowledgement are fenced by the current unexpired token.
- Runtime pods keep auto-migration disabled. Production applies migrations through the release migration job.
